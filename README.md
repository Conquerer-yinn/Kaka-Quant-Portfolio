# Kaka_Quant

Kaka_Quant 是一个面向私募交易团队研究与执行流程的量化工作台，核心目标不是单纯做看盘工具，而是把交易员日常依赖的大量重复工作系统化，包括盘中观察、盘后复盘、指标研究、策略研究、轻量回测、信号推送和日常执行管理。

这个项目最初的出发点，是把一线交易流程里高度依赖人工经验和临场状态的部分尽量自动化、结构化，让交易员既能更快理解盘面，也能把成熟策略稳定执行下去，降低情绪波动、主观偏差和人工遗漏带来的操作变形。

对外展示时，这个项目强调的是两件事：

- 它是一套真实的量化研究工作流，而不是静态展示项目。
- 它也体现了复杂工程项目的落地能力，包括数据链路、服务封装、前端展示、消息推送和设计沉淀。

## 核心技术能力

- 量化研究工作流设计
- 多源数据接入与整合
- Python 自动化任务编排
- FastAPI 服务化封装
- React 数据可视化展示
- 飞书工作流与消息推送集成
- Docker 容器化部署
- 轻量研究系统架构设计
- 研究结果沉淀与执行流程标准化

实现层主要依赖：

- Python / Pandas
- FastAPI
- React / Vite
- Docker / Docker Compose / Nginx
- Excel 工作簿自动化
- Feishu Bot Webhook
- Tushare / 问财数据接入

## 项目定位

项目由两大核心模块组成：

### 1. 指标数据研究模块

这一部分服务于交易员对盘面行情的理解与执行辅助，重点包括：

- 盘中观察
- 盘后复盘
- 市场情绪与关键指标研究
- 历史数据维护与结构化沉淀
- 飞书卡片推送

它的作用不是直接替代交易判断，而是把高频、重复、容易遗漏的盘面信息整理成更清晰的结构，帮助交易员在盘中和盘后更稳定地理解市场。

### 2. 策略研究与执行模块

这一部分服务于交易策略的系统化落地，重点包括：

- 策略想法整理与规则沉淀
- 历史样本筛选
- 轻量回测与夜间复盘
- 策略表现记录与对比
- 成熟策略的日常信号推送
- 执行流程标准化

这条主线的核心价值，是把“策略”从个人经验和临场发挥，逐步转成可回看、可复盘、可推送、可持续执行的规则系统，减少因情绪影响导致的误判和执行偏差。

## 项目亮点

- 围绕私募交易团队的真实工作流设计，而不是为了展示而虚构页面或数据。
- 用统一历史主表维护研究结果，兼顾补数、复盘、模板复用和历史回看。
- 通过 FastAPI 把数据任务、展示接口和消息接口统一服务化。
- 通过 React 前端把研究结果、历史数据和任务状态做成可直接演示的可视化界面。
- 打通从数据获取、规则整理、结果生成到飞书触达的完整工作流，既服务指标研究，也服务策略信号推送。
- 为策略研究、轻量回测和成熟策略信号推送保留清晰的结构基础，具备继续扩展成完整策略工作流的能力。
- 提供完整 Docker 部署方案，支持前后端分容器部署、Nginx 统一入口、环境变量注入与数据卷挂载。
- 保留 `project_memory/` 作为设计沉淀系统，用来记录结构约定、架构说明和关键设计决策，体现工程抽象和长期维护意识。

## 能力地图

### 1. 指标研究与历史复盘

项目当前围绕 `market-sentiment` 维护统一历史主表：

- 历史数据按固定命名规则存放在 `storage/data_master/`
- 增量更新时先生成补充数据，再并回历史主表
- 前端历史页固定读取最新历史主表中的最近交易日数据
- 保留 Excel 作为研究复盘、指标观察和图表承载的重要数据载体
- 将工作簿更新、结果整理、模板写回视为可调用的工作流节点，便于继续接入 MCP / skill 型工具来统一调度数据写回与结果产出

当前展示示例文件：

- [历史数据_20260101_20260402.xlsx](d:/code/Kaka_Quant_Portfolio/storage/data_master/%E5%8E%86%E5%8F%B2%E6%95%B0%E6%8D%AE_20260101_20260402.xlsx)

### 2. 服务化任务运行

后端基于 FastAPI 构建，负责把脚本任务和展示能力封装成稳定接口。

主要能力包括：

- 健康检查与任务列表
- `daily-basics` 与 `market-sentiment` 任务触发
- `market-sentiment` 后台任务状态轮询与取消
- 首页概览、历史数据、卡片页所需接口
- 三类飞书卡片的刷新与发送接口

### 3. 前端可视化展示

前端基于 React + Vite 构建，强调真实数据展示而不是静态页面堆砌。

当前页面包括：

- `/` 项目首页
- `/market/history` 历史数据页
- `/market/push` 推送卡片页
- `/strategies` 策略页

页面特性包括：

- 图表与表格共用同一份后端数据
- 支持历史任务发起、轮询、取消
- 支持卡片预览、刷新、发送
- 保留轻量展示风格，便于快速理解项目结构与数据流

### 4. 飞书工作流与结果推送

飞书推送链路在这个项目里不是单独的消息模块，而是整个研究与执行工作流的触达层。

当前已经落地的公开展示能力包括：

- 盘后复盘卡片
- 竞价观察卡片
- 盘中节奏卡片

而这套链路本身的设计目标并不局限于上述三类市场卡片，它同样服务于策略模块，能够承接：

- 指标研究结果推送
- 复盘结果推送
- 成熟策略信号推送
- 日常执行结果推送

推送链路采用“数据源 -> 计算与整理 -> 快照服务 -> 视图生成 -> 飞书触达”的方式组织，重点不是单发一条消息，而是把研究结果和执行结果稳定送达到指定角色和指定机器人。

### 5. 策略研究、轻量回测与执行

策略模块是这个项目非常重要的一部分，不是附属功能。

它的整体设计可以分成三层：

#### 研究层

- 整理策略逻辑和触发条件
- 记录样本筛选口径
- 沉淀复盘结论和迭代方向

#### 回测层

- 通过轻量脚本和 Excel 结果组织进行夜间回测
- 对开发中的策略做样本验证、结果对比和参数调整
- 保留一个简易回测系统，而不是上来就引入重型回测框架

#### 执行层

- 将成熟策略纳入日常运行清单
- 固定输出信号结果
- 按策略配置推送到指定飞书机器人
- 降低临盘执行中的主观偏差和情绪干扰

换句话说，这个项目既服务于“研究”，也服务于“执行”；既帮助交易员理解市场，也帮助团队把交易纪律更稳定地落到系统中。

### 6. 知识增强复盘系统

除了结构化指标和固定流程任务，这个项目还规划了一套 `知识增强复盘系统`。

在私募交易团队的真实工作环境中，很多高价值信息并不只存在于当天行情数据里，还沉淀在历史复盘文档、策略文档、研究笔记、公众号技术推文存档等资料中。这些内容本质上构成了一套团队长期积累下来的研究语境和判断框架。

这套能力的目标不是让模型脱离数据自由生成结论，而是：

- 先基于固定流程产出当天的客观数据
- 再检索历史复盘、策略文档和研究资料中的相关内容
- 最后结合“当天数据 + 历史知识”生成更贴近团队研究风格的文字复盘、小结和解释

它可以服务的场景包括：

- 盘后复盘小结生成
- 指标异动解释
- 策略信号背景说明
- 历史相似案例参考
- 研究结论调用与复用

如果从技术视角理解，这部分可以看作是一种面向量化研究场景的检索增强工作流，但项目更强调的是业务落地价值，也就是让系统输出不仅有数据，还具备团队内部一致的理解框架和表达语境。

### 7. 设计沉淀与文档系统

项目中的 `project_memory/` 不是简单的内部笔记目录，而是一套面向工程协作和长期维护的设计沉淀系统。

这里记录的内容包括：

- 结构设计与模块边界
- 数据流与服务分层约定
- 典型问题说明
- 项目状态摘要
- 主题归档摘要

对于公开仓库来说，这部分很有价值，因为它能让阅读者更快看懂这个项目不是“脚本集合”，而是一套有明确结构、有设计表达、有长期维护意识的工程化系统。

## 工作流自动化

这个项目已经把研究与执行流程中的多段人工操作整理成可复用的自动化链路，重点包括：

- 通过 Tushare、问财等外部数据入口获取研究所需原始数据
- 对多源数据进行清洗、整合、加工，并沉淀为统一历史主表和任务输出
- 将历史维护、指标计算、卡片生成、结果分发封装成可重复运行的任务流程
- 将结果同步输出到 Excel、前端页面和飞书机器人等不同消费端
- 将策略研究、轻量回测、信号推送和执行记录逐步纳入同一套工作流体系
- 将工作簿更新、复盘材料整理、结果写回等步骤抽象成工具调用节点，便于继续通过 MCP / skill 型工具统一调度

这套自动化能力覆盖的不是单点脚本，而是从数据接入、结果计算、复盘整理到消息触达的整条链路。

## 运行方式

### Docker 部署

项目提供完整的容器化部署方案：

- `backend` 容器负责 FastAPI 接口、数据任务和推送链路
- `frontend` 容器负责 React 静态资源托管
- `nginx` 作为前端容器内统一入口，并反向代理 `/api`
- `storage/` 通过数据卷挂载，保留历史主表和备份文件
- 配置项通过 `.env` 注入，避免把敏感信息写入镜像

使用方式：

```bash
cp .env.example .env
docker compose up --build -d
```

启动后访问：

- 前端首页：`http://127.0.0.1`
- 后端 API：`http://127.0.0.1:8000`
- 统一代理入口：`http://127.0.0.1/api`

项目同时提供了更正式的命令入口：

- `Makefile`：统一常用开发和部署命令
- `scripts/dev-backend.ps1`：启动后端开发服务
- `scripts/dev-frontend.ps1`：启动前端开发服务
- `scripts/dev-full.ps1`：同时打开前后端开发进程
- `scripts/docker-up.ps1`：启动完整容器化部署
- `scripts/docker-down.ps1`：停止容器化部署

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

前端依赖安装：

```bash
cd frontend
npm install
```

### 2. 配置环境

项目至少依赖以下配置：

- `TUSHARE_TOKEN`
- `FEISHU_BOT_WEBHOOK`
- `TUSHARE_REQUEST_DELAY` 可选

### 3. 启动后端

```bash
uvicorn backend.main:app --reload
```

默认地址：

- API：`http://127.0.0.1:8000`
- Docs：`http://127.0.0.1:8000/docs`

### 4. 启动前端

```bash
cd frontend
npm run dev
```

默认地址：

- Frontend：`http://127.0.0.1:5173`

如果你希望使用统一命令入口，也可以直接运行：

```powershell
./scripts/dev-backend.ps1
./scripts/dev-frontend.ps1
./scripts/dev-full.ps1
./scripts/docker-up.ps1
```

## 接口设计

整个项目的接口围绕四类能力组织：任务运行、研究展示、策略工作流、结果推送。

### 系统与任务接口

- `GET /health`
- `GET /tasks`
- `POST /tasks/daily-basics/run`
- `POST /tasks/market-sentiment/run`
- `GET /tasks/market-sentiment/{task_id}`
- `POST /tasks/market-sentiment/{task_id}/cancel`

### 指标研究与展示接口

- `GET /dashboard/summary`
- `GET /market/history/market-sentiment`
- `GET /strategies`
- `GET /strategies/{strategy_id}`

### 策略执行与信号接口

- `GET /strategies/signals/today`
- `POST /strategies/{strategy_id}/signals/generate`

### 飞书推送与结果分发接口

- `GET /market/push/cards`
- `POST /market/push/post-close/refresh`
- `POST /market/push/post-close/send`
- `POST /market/push/auction/refresh`
- `POST /market/push/auction/send`
- `POST /market/push/intraday/refresh`
- `POST /market/push/intraday/send`
- `POST /strategies/{strategy_id}/push/preview`
- `POST /strategies/{strategy_id}/push/send`

## 设计取舍

- 不直接做重型量化平台，而是优先围绕真实研究与执行流程构建高可用的小型工作台。
- 保留 Excel 作为研究载体，同时补上 API 与前端展示层，兼顾实用性和作品展示效果。
- 前端坚持消费真实数据，而不是使用演示假数据，保证项目具备可信度。
- 卡片推送链路与历史展示链路分层实现，避免后续扩展时相互牵制。
- 策略模块优先走轻量回测、研究沉淀和信号推送路径，再逐步扩展成更完整的日常执行体系。

## 后续发展方向

- 继续完善策略研究与回测执行链路，把样本筛选、回测统计、结果复盘和信号推送进一步标准化。
- 将成熟策略逐步接入更稳定的日常执行体系，并补充更细的策略注册、调度和结果跟踪能力。
- 进一步完善知识增强复盘系统，把历史复盘文档、策略文档和研究资料更系统地纳入检索增强流程。
- 在执行层继续预留量化交易接口接入能力，例如 QMT 等实盘或半自动执行接口，逐步从研究与推送延伸到更完整的交易执行闭环。
