# 市场字段设计文档

## 文档目的

这份文档用于统一管理 `market/` 下的字段设计，重点解决四件事：

1. 某个字段是做什么的。
2. 这个字段的数据从哪里来。
3. 这个字段在代码里已经实现了没有。
4. 这个字段会用于历史 Excel、飞书卡片，还是两者都会用。

## 当前推荐分层

当前 `market` 相关字段建议按三层理解：

1. 原始数据层
   来自 Tushare、竞价采集、盘口采集、龙虎榜等原始数据。
2. 指标计算层
   基于原始数据计算出的市场指标、十日高度、创业板反馈等。
3. 输出视图层
   面向不同输出场景组织字段。
   - Excel 视图
   - 飞书卡片视图

注意：

- 同一个字段可能同时服务 Excel 和飞书卡片。
- 飞书卡片不是另一套底层数据，只是“推送展示视图”。

补充约定：

- 盘后飞书卡片应直接基于原始数据层和指标计算层生成，不应反向从 Excel 视图层取数。
- Excel 视图和飞书卡片视图都属于输出视图层，但两者应分别从下层指标结果构造。

## 字段记录规则

### 1. 已实现字段

对于已经写进代码的字段，文档必须记录：

- 真实 Excel 列名
- 所属模块
- 代码锚点
- 口径说明
- 使用位置

这里的“代码锚点”优先使用当前真实常量或真实列名来源，例如：

- `DAILY_BASICS_COLUMNS`
- `MARKET_OVERVIEW_COLUMNS`
- `HEIGHT_OBSERVATION_COLUMNS`
- `CHINEXT_COLUMNS`
- `CHINEXT_FEEDBACK_COLUMNS`

### 2. 规划字段

对于还没有在程序里落地、但已经需要设计的字段，文档里统一标记为：

- `状态：规划中`
- `代码锚点：待实现`

这样可以先把字段定义清楚，后面再按文档补程序。

### 3. 关于“字段编码”

你前面指出的问题是对的：如果文档里的字段编码和代码里真实字段对不上，后面维护会很乱。

因此统一这样处理：

1. 对于已实现字段：
   - 文档主键使用稳定的英文编码，便于后续程序化管理。
   - 同时必须记录真实 Excel 列名。
   - 同时必须记录真实代码锚点。
2. 对于未实现字段：
   - 先定义稳定的英文编码。
   - 明确标记“规划中”，不冒充已经落地。

### E. `market_sentiment` -> 位置度量辅助字段

来源文件：

- `market/indicators/position_metrics.py`

说明：

- 这些字段不是独立原始指标，而是给某些数值字段附加的辅助判断列。
- 生成规则为：`原字段名 + 位置`、`原字段名 + 相对中枢`。

例如：

- `总成交额(亿元)位置`
- `总成交额(亿元)相对中枢`
- `全市场近十日高度(%)位置`
- `昨日创业板涨停股次日收盘溢价(%)相对中枢`

当前用途：

- 主要服务 Excel 总览摘要区。
- 不建议大量直接进入飞书卡片。

### F. `market_sentiment` -> `总览`

说明：

- `总览` 是展示层，不是权威历史数据源。
- 它由 `run_market_sentiment.py -> build_overview_rows()` 从各个数据 sheet 抽取和拼接。
- 因此后续如果出现数字错误，应优先回查：
  - `总市场数据`
  - `高度观察`
  - `创业板专区`

## 三、当前最重要的结构结论

### 1. `daily_basics` 当前可以弱化，但不建议直接删除

原因：

1. 它是最稳定的基础日度历史表。
2. 虽然有些内容在 `market_sentiment` 里已经被包含，但它仍然适合作为“最基础的 demo / 基础历史数据表”保留。
3. 后续如果你确认完全不需要它，再考虑逐步淡化，而不是现在就强行移除。

### 2. 当前更推荐的结构不是“历史代码”和“推送代码”两堆硬拆

更合理的是：

1. 原始数据层
2. 指标计算层
3. 输出视图层
   - Excel 视图
   - 飞书卡片视图

这样后续盘后、盘中、竞价推送都可以共用一部分底层逻辑，避免重复维护。

### 3. 后续写程序时的推荐顺序

1. 先把 `盘后复盘卡片` 的推送字段做成可用版本。
2. 再补 `竞价观察卡片` 所需的原始数据链路。
3. 最后补 `盘中节奏卡片` 的采集和推送逻辑。

原因：

- 盘后卡片的字段来源最明确，基本都能从现有 `market_sentiment` 里抽。
- 竞价和盘中字段目前还缺数据采集链路，先定义字段、后补任务更稳。
